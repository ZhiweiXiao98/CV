较早使用 CAD 绘制模型零部件，并通过激光切割完成建筑模型制作，开始关注工具对效率和精度的提升。

工作阶段

03

AI 工作方式

日常使用 AI 时，重点不在“提问”，而在如何组织任务、管理记忆、调度模型、设计工具接口，并持续优化 Agent 的执行效率和用户体验。

Planning

先成计划

复杂任务会先交给 AI 做分析和拆解，按阶段、步骤、依赖关系输出计划书，再进入执行。这样可以先确定任务边界、风险点和验证方式，而不是直接开始生成代码。

Context Management

记忆分层

上下文分为长期记忆、中期记忆和短期记忆：长期记忆包含用户偏好、系统提示词、动态加载的 Skills 和工作日志；中期记忆保存压缩后的上下文摘要，并在需要时包含当前正在执行的任务书；短期记忆承载当前任务上下文。

Model Routing

模型路由

多模型协作更接近路由系统：由主 Agent 根据任务复杂度、模型能力和所需 Skills 分配给不同模型。不同模型调用工具的习惯差异很大，需要针对模型设计不同 Harness。

Evaluation

交叉校验

重要问题会交给多个不同 AI 分析，或把一个 AI 的输出交给其他 AI 审查。执行遇到困难时，会根据模型工作习惯调整提示词、工具设计、上下文组织和执行流程。

Optimization

压缩损耗

Agent 工作流不只看能不能完成任务，也要关注延迟、通信、思考、工具调用和反馈效率。需要持续优化每一次模型调用、工具返回和上下文压缩带来的时间成本。

Product Experience

从可用到顺手

AI Bridge 不只追求功能可用，也关注界面布局和使用体验。选择 PySide 架构，是希望像 Adobe 工具一样提供更美观、高度自定义的工作区，让 AI Coding 工具更适合长时间使用。

04

不止使用工具

使用 Codex、Claude Code、Trae 等 AI Coding 工具时，不只关注它们能做什么，也会观察它们如何组织上下文、调用工具、处理错误、适配模型，以及哪些环节不完全适合自己的工作方式。

Tool Analysis

先理解工具，再重做工具。

现成工具提供了很好的参照，但上下文、模型路由、界面布局、工具调用和工作日志沉淀这些环节，仍然需要更贴合个人工作流的设计。

可控的链路自研核心链路能更清楚地理解 Agent 运行机制，也方便在出错时定位和溯源。
拆到关键处核心链路没有直接套用 LangChain，也没有直接基于 Monaco Editor 和 Electron 搭建，而是尽量拆解并实现关键模块。
适配不同模型不同模型调用工具的习惯差异明显，需要围绕模型能力和工具行为做适配。
05

AI Bridge

AI Bridge 是这种工作方式的工具化沉淀：面向个人开发和复杂任务推进场景，支持大模型通过工具调用、知识检索、代码执行、文件操作、上下文管理和反馈机制参与软件开发流程。

Why Build

需求长出工具

在长期使用 AI 进行开发和任务推进时，普通聊天界面和现成 AI Coding 工具并不能完全覆盖上下文、工具、日志、布局和路由需求，于是开始把自己的工作流沉淀成 AI Bridge。

Agent Workflow

任务流转

计划、执行和验收不要求由同一个 Agent 完成；不同阶段可按任务复杂度路由到不同模型和 Skills。

01
需求输入用户提出开发、整理、检索、改造或排查需求。
02
资料收集AI 读取背景、项目结构、历史上下文，并追问缺失细节。
03
计划书生成按阶段、步骤、依赖、风险和验收方式输出任务计划。
04
计划对齐用户提出修改建议，AI 继续调整计划，直到双方对齐。
05
进入执行AI 按计划逐项推进，执行文件修改、检索、代码运行或工具调用。
06
成果校验检查计划书中的事项是否完成，并结合测试、日志和工具反馈判断结果。
07
AI 验收验收 Agent 审查执行结果；未通过则反馈问题，继续进入执行循环。
08
用户验收AI 验收通过后交付用户，由用户确认结果或提出新的调整意见。
09
日志归档归档工作日志、关键决策、问题记录和可复用经验，进入后续长期记忆。
规划 Agent 可以负责分析和任务书，执行 Agent 负责落地，验收 Agent 负责审查结果；完成后的工作日志会归档到长期记忆，成为下一轮任务的上下文资产。

06

系统骨架

项目围绕模型调用、工具执行、上下文管理、反馈机制、运行环境和多端交互展开，重点验证 Agent 工作流在个人开发场景中的可用性。

ServerFastAPI + WebSocket 中台，负责连接管理、消息分发、认证、状态同步和服务编排。
Client / MobilePySide6 桌面端与 Flet 移动端，支持远程连接、会话管理、消息展示和工具结果呈现。
LLM Sources浏览器模式从 Web AI 页面抓取信息，API 模式从模型接口获取输出，并支持多模型 Profile 与 Fallback Chain。
Tools / PluginsSkills、Docker 沙箱、知识检索、网络搜索、面板插件、Rhino 集成和后台建议助手。
RAG

知识索引

基于 ChromaDB + fastembed 实现向量检索能力，支持增量索引、重排序、分块策略和知识库任务管理，让 Agent 能检索项目资料和历史上下文。

Tool Execution

工具运行时

基于可扩展 Skill 插件架构，内置文件操作、代码执行、知识搜索、网络搜索，并通过工具运行时处理调用、权限、结果和异常。

Context

上下文回路

设计系统、长期、工作、短期多层上下文窗口，支持自动压缩；后台建议助手可基于当前上下文提供纠错建议和下一步思路。

Sandbox

隔离执行

通过 Docker 提供隔离的代码执行环境，配合工具路由和执行历史，降低用户代码运行风险并增强调试可追踪性。

Browser Mode

浏览器消息源

通过 Selenium 驱动 Chrome 与 Web AI 页面交互，包含消息解析、输入发送、会话同步和浏览器状态机。

Extensibility

外部扩展

包含面板插件架构、Skills 管理面板、Rhino 监听器与插件入口，展示系统面向外部工具和专业软件工作流的扩展能力。
